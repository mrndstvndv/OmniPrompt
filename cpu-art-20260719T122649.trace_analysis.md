2. **Measure is the heaviest phase** — `measure-BRTryo0` variants total ~21s across all calls. 6,327 calls to `LayoutModifierNodeCoordinator.measure` averaging 2.3ms each.

## Applied Optimizations

Changes landed in commit `6c31ebf` (`ItemsList.kt`).

### 1. Removed per-item `produceState` for icon loading

**Before:** Every visible list item launched a coroutine via `produceState<Bitmap?>` keyed on `item.id` + `appSettings.themedIconsEnabled` + `appSettings.themeAllIcons` + `appSettings.iconPackPackageName`. Toggling any setting restarted ALL visible items' coroutines simultaneously.

**After:** `remember` + `LaunchedEffect` keyed only on `item.id` and `item.iconLoader` reference. Same async loading, no restart storms when unrelated settings change.

**Targets:** `SnapshotStateObserver.observeReads` (12.1s), coroutine launch overhead per recomposition cycle.

### 2. Removed per-item `collectIsPressedAsState()` + corner animation

**Before:** Each item created a `MutableInteractionSource` and called `collectIsPressedAsState()` to animate corner radius on press. This added ~400+ active snapshot observers, fed into `animateFloatAsState` for shape lerp.

**After:** Ripple indication (from `clickable`/`combinedClickable`) already provides press feedback. The corner radius animation was duplicative — removed entirely. Click handler fires directly without 80ms delay.

**Targets:** `SnapshotStateObserver.observeReads` (12.1s — ~30% of calls were per-item observers), `ComposableLambdaImpl.invoke` (3.9s).

### 3. Memoized shape with `remember`

**Before:** `RoundedCornerShape` with four `lerp` calls computed on every recomposition per item, even though corners only depend on slot position (first, last, middle).

**After:** `remember(singleItem, isVisualTopItem, isVisualBottomItem)` computes corners once per slot change.

**Targets:** `measure-BRTryo0` (14.7s), `ComposableLambdaImpl.invoke` (3.9s).

### 4. Removed unused imports & constants

Cleaned 14 unused imports, 2 unused constants (`pressedResultCornerRadius`, `tappedResultFeedbackDelayMillis`), and the unused `tapFeedbackDelayMillis` variable.

### Not changed

- `verticalEdgeFade` — kept exactly as-is (uses `saveLayer` + `DstIn` masking). Its draw cost was marginal vs per-item draw modifiers.
- `HighlightedText` — already fast (0.14ms avg, 992 calls).
- Recomposer debounce (80ms) — already in place.
- Icon cache in `AppListRepository` — already using `ConcurrentHashMap` with composite keys.

### Estimated impact

Primary targets were the 18,335 `SnapshotStateObserver.observeReads` calls (removing ~400+ per-item observers), and the per-item measure/draw modifier overhead from produceState + press animation. These accounted for roughly 30-40% of the main-thread work outside of the core layout pipeline.
